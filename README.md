# TON Wallet — Testnet Self-Custodial Wallet

Self-custodial кошелёк для TON testnet. Полностью клиентское приложение — приватные ключи генерируются, шифруются и хранятся исключительно в браузере пользователя.

```bash
npm install

## Архитектура

Приложение построено по модели **"толстый клиент + публичный RPC"** — той же, что используют Tonkeeper, MetaMask и Trust Wallet. Вся криптография, формирование транзакций и управление состоянием выполняются внутри браузера. Единственная внешняя зависимость — публичный API TON Center, который служит точкой входа в блокчейн для чтения данных (баланс, история) и трансляции подписанных транзакций (BOC).

```
┌──────────────────────────┐           ┌───────────────────────────┐
│   Браузер (React SPA)    │── HTTP ──▶│  TON Center API (Testnet) │
│                          │           │                           │
│  • Генерация ключей      │           │  • Баланс                 │
│  • Подпись транзакций     │           │  • История транзакций     │
│  • Шифрование (AES-GCM)  │           │  • Трансляция BOC         │
│  • addressGuard          │           └───────────────────────────┘
├──────────────────────────┤                       │
│  localStorage            │                       ▼
│  (зашифровано PIN-кодом) │           ┌───────────────────────────┐
│  • Мнемоника             │           │  TON Blockchain (Testnet) │
│  • Доверенные адреса     │           └───────────────────────────┘
│  • История адресов       │
└──────────────────────────┘
```

Секретная фраза (мнемоника) шифруется ПИН-кодом пользователя через цепочку `PBKDF2 (100K итераций) → AES-256-GCM`. При отправке транзакции ПИН расшифровывает мнемонику в оперативной памяти, SDK формирует `KeyPair`, подписывает сообщение, отправляет BOC — и ключ сразу уничтожается. На диск или в сеть приватный ключ не попадает никогда.

### Обоснование стека

| Технология | Обоснование |
|---|---|
| **React 18 + TypeScript + Vite** | SPA-архитектура принципиальна: SSR (Next.js) означал бы, что ключи потенциально попадают в серверную память. Vite обеспечивает мгновенный HMR и нативную поддержку ESM. TypeScript — строгая типизация критична для финансового приложения |
| **@ton/ton + @ton/core + @ton/crypto** | Официальный SDK от разработчиков TON. Типизирован, активно поддерживается, поддерживает WalletV4R2. Альтернатива — `tonweb` — устарела, не типизирована, не поддерживает современные SendMode |
| **Web Crypto API (AES-GCM + PBKDF2)** | Нативная браузерная криптография. Не тянем сторонние npm-пакеты с потенциальными уязвимостями в supply chain. PBKDF2 со 100K итераций делает брутфорс PIN-кода экономически нецелесообразным |
| **Vitest** | Нативная интеграция с Vite, в 2–5 раз быстрее Jest на проектах с ESM |

```

Разделение на `services/` и `components/` — осознанное: бизнес-логика не зависит от React и может быть покрыта unit-тестами без рендеринга компонентов.

## Логика проверок: защита от Address Spoofing

Центральный модуль безопасности — `addressGuard.ts`. Он выполняет **5-этапный анализ** адреса получателя до формирования BOC-транзакции:

| # | Проверка | Механизм |
|---|---|---|
| 1 | **Address Poisoning** | Сравнение prefix/suffix (первые 4, последние 4 символа) с ранее использованными адресами. Если крайние символы совпадают, а центральная часть — нет, это классическая атака poisoning → `critical` |
| 2 | **Clipboard Tampering** | При вставке (`Ctrl+V`) сверяет вставленный текст с текущим содержимым `navigator.clipboard`. Расхождение говорит о том, что малварь подменила адрес в буфере → `critical` |
| 3 | **Uninitialized Contract** | Асинхронная проверка через API — активирован ли смарт-контракт получателя. Отправка на неинициализированный bounceable-адрес приведёт к возврату средств (с вычетом комиссии) |
| 4 | **Zero Balance** | Если контракт не инициализирован и баланс = 0, дополнительное предупреждение — возможно, адрес никогда не использовался |
| 5 | **Whitelist** | Если адрес в списке доверенных — все эвристики пропускаются |

Дополнительные проверки:
- **Self-transfer** — программная блокировка отправки на собственный адрес
- **Large amount** — предупреждение при отправке > 50% баланса
- **Network mismatch** — блокировка mainnet-адресов
- **UI-контур** — при любом предупреждении кнопка "Отправить" заблокирована до активации чекбокса "Я понимаю риски"

## Компромиссы

| Решение | Обоснование | Production-альтернатива |
|---|---|---|
| **localStorage** | Синхронный, простой, достаточен для scope задания. Мнемоника хранится зашифрованной (AES-256-GCM), не в открытом виде | `IndexedDB` + изоляция крипто-операций в `Web Worker` — XSS-скрипт из основного потока не сможет прочитать память воркера |
| **PIN-код** | Привычный UX, быстрый ввод. PBKDF2 100K итераций компенсирует малый keyspace | `WebAuthn` (FaceID/TouchID) — и удобнее, и безопаснее |
| **Polling (10 сек)** | TON Center testnet не предоставляет WebSocket-эндпоинт | `tonapi.io` SSE/WebSocket — push-обновления баланса за миллисекунды |
| **Non-bounceable** | Безопаснее для p2p-переводов: средства не "отскочат" обратно, если контракт получателя не инициализирован | Автоопределение bounce-флага по `isInitialized` контракта |
| **Без API-ключа** | TON Center принимает анонимные запросы (1 req/sec) — хватает для тестирования и демо | API-ключ через `@tonapibot` снимает rate-limit |
| **Truncation баланса** | `formatTon` строго отсекает десятичные знаки, а не округляет. 1.9999 TON → "1.9999", не "2.00". Это гарантирует, что пользователь никогда не увидит баланс больше реального и не получит ошибку "Insufficient funds" | — |
| **SendMode 128 (MAX)** | Кнопка MAX делегирует расчёт комиссии блокчейну через нативный `CARRY_ALL_REMAINING_BALANCE`. Это точнее, чем ручной `balance - estimatedFee`, и устойчиво к изменению gas-модели | — |

## Дальнейшие улучшения

### Безопасность
- **Web Workers для изоляции ключей** — вынести `mnemonicToPrivateKey`, подпись транзакций и `deriveKey` в отдельный Worker. Основной JS-поток (и любой XSS-скрипт в нём) физически не сможет получить доступ к памяти воркера. Это та же модель, что использует 1Password в браузерном расширении.
- **WebAuthn (FIDO2)** — заменить PIN на биометрическую аутентификацию через платформенный аутентификатор (Secure Enclave на Apple, TPM на Windows). Ключ деривации привязан к аппаратному модулю и не извлекается.
- **CSP и Subresource Integrity** — настроить строгий `Content-Security-Policy` и SRI-хеши для всех подгружаемых скриптов, чтобы исключить инъекцию через CDN-компрометацию.

### Функциональность
- **TON Connect 2.0** — интеграция протокола для подключения кошелька к DApp-экосистеме (DEX-свапы, NFT-маркетплейсы, стейкинг). Технически это `bridge`-сервер + session key exchange через QR-код.
- **Jettons и NFT** — расширить `walletService` для чтения балансов Jetton-контрактов (TEP-74) и метаданных NFT (TEP-62). Потребует дополнительных RPC-вызовов `runGetMethod` к мастер-контрактам токенов.
- **Multi-account (HD Wallet)** — деривация нескольких аккаунтов из одной мнемоники по стандарту BIP-44. Позволяет разделять средства без создания отдельных seed-фраз.

### Инфраструктура
- **E2E-тесты (Playwright)** — полное покрытие user-flow: создание кошелька → отправка → проверка истории. Критично для регрессионного тестирования UI-зависимой логики (clipboard, copy, модалки).
- **PWA (Service Worker + Manifest)** — установка как нативное приложение с офлайн-кешированием статики.
- **Мониторинг** — интеграция Sentry для отлова runtime-ошибок в production (особенно важно для `crypto.subtle` совместимости между браузерами).

## Тесты

```bash
npm test    # 47 тестов, 3 сьюта, ~1.5 сек
```

| Файл | Покрытие |
|---|---|
| `format.test.ts` | Форматирование TON (проверка truncation vs rounding), валидация адресов (testnet/mainnet/raw), парсинг пользовательского ввода суммы |
| `addressGuard.test.ts` | Полный оркестратор `checkAddress`: 9 сценариев (invalid → mainnet → self-transfer → trusted bypass → new address → large amount → full balance → uninitialized → network error). Детекция address poisoning: 5 кейсов |
| `storageService.test.ts` | CRUD доверенных адресов (add/remove/deduplicate), история использованных адресов, проверка существования кошелька |

Функции, зависящие от блокчейна (`createWallet`, `sendTon`, `getTransactions`) и Web Crypto API (`encryptAndStore`, `decryptAndLoad`) — не покрыты unit-тестами намеренно: это интеграционные вызовы, которые корректно тестируются через E2E (Playwright).
